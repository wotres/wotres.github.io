깃이용방법

#깃(Git) - 형상관리 툴(분산 버전 관리 시스템)

##사용 이유 및 설치하고 사용하는 명령어 중심으로 기술하였다.

##사용 이유 :

  • 중앙 집중식 버전 관리 시스템은 서버에서 적절한 백업이 수행되지않으면 서버 소실시 자료도 모두 소실
    but, DVCS(Distributed Version Control System)는 각각 로컬 저장소가 독립된 서버 역할을 수행
  • 중앙 집중식 시스템의 경우 트래픽 집중으로 속도 저하의 우려와 서버 장애시 작업 자체가 불가능한 단점
    but, DVCS는 서버 장애와 무관하게 로컬 저장소에 대한 자유로운 작업이 가능

##설치 :
https://git-scm.com/ 에 접속 후 해당 운영체제에 맞는 버젼 설치

Git Bash는 리눅스/유닉스 쉘을 사용할 수 있도록해주는 프로그램

확인 명령어

git – version : 깃 버전 보여줌(깃 설치 되었는지 확인)

##깃허브에서 프로젝트 생성하는법

  • https://github.com/ 접속
  • start a project 클릭후 저장소 생성 및 Repository name 에서 저장소 이름 설정
  • 명령순서
    깃에 사용자를 알려줌(파일이 누구의 것인지 정보를 달수있음)
    git config –global user.name “유저이름”
    git config –global user.email “유저 이메일”
    git init // 현재 폴더에 git 저장소를 생성함
    해당폴더에 파일 ex) python 파일 생성
    git status // 해당폴더 아래 파일이 반영되었는지 상태를 알려줌
    git add 생성파일.py // 해당 파일 추가
    git add . // 전체 add
    git commit -m “설명” // 깃에 반영 및 설명 부분에 변경사항에 대한 메시지 작성 가능
    // 반영됨

깃허브에서 브랜치 생성 및 반영하는법

  • 명령순서
    git branch 브랜치명 // 브랜치명 생성
    git branch // 현재 생성된 브랜치와 활성화된 브랜치를 확인한다.
    git checkout 원하는브랜치명 // 원하는 브랜치 명으로 활성화
    해당폴더에 파일 ex) python 파일 생성
    git add -A // 깃에 모든 파일 반영
    git commit -m “설명” // 반영된 파일 설명 및 깃 반영
    git push origin 브랜치명 // 브랜치에서 반영된 정보를 깃에 반영한다.
    깃허브 홈페이지에서 해당 브랜치명으로파일 올라온것 확인후 Compare & pull request 버튼 클릭
    깃 머지 실행
    git checkout master // master로 이동
    git pull // pull 명령어로 로컬저장소를 원격 저장소와 같게 만든다. 기준은 원격저장소
    위 pull에서 조심해야하는게 작업해놓은게 있는데 커밋하지않고 마스터로 이동한뒤 pull 땡기면 파일 사라짐
    git merge 브랜치명 // 병합작업 수행
    이부분에서 오류 잘남 why? 서로 다른 깃내려받은뒤 작업 및 반영 시점에 따라 다를 수 있기 때문
    충돌날 경우 파일 수정 작업 수행
    git status // 상태확인 후
    git add -A // 반영
    git push origin master // 마스터를 push 하여 원격 저장소에 업로드함\
    git branch -d 브랜치명 // 해당 브랜치 삭제

깃허브에 있는 내용 내 로컬 저장소로 가져오는법

  • 명령순서
    git clone 깃주소 // clone을 이용해 생성한 원격저장소를 로컬 저장소로 파일 복사
    복사가 아니라 바로 연결해서(내파일이 있는경우) 올려야하는 경우에는
    git remote add origin 깃주소 // 원격 저장소를 추가 , origin은 추가하는 원격저장소의 이름
    git remote -v // 원격저장소와 연결되었는지 확인
    git push origin -all // 원격저장소 별칭(여기선 origin) all은 oring저장소의 모든 로컬 브랜치를 푸시함
    파일생성
    git add 생성한파일
    git commit -m “설명”
    git push origin master // 원격저장소 와 마스터 브랜치 모두 push 하면 깃허브에 반영된것 확인

원격 저장소와 로컬 저장소의 버젼을 맞추는법(병합하는법)

git pull과 git fetch 두개가 있는데 pull 명령어는 원격 저장소의 정보를 가져오면 자동으로 로컬 브랜치에 병합까지 수행
따라서 어떤 점이 변했는니 알아보기 힘들 수 있어 주로 fetch 사용

  • 명령어
    원격저장소 로컬파일 모두 수정 // github에서 수정은 원격저장소에서 commit랑 같음
    git commit -a -m “설명” // 로컬에서 커밋
    git push origin master // 로컬 브랜치의 내용을 origin 원격저장소에 push
    여기서 오류 발생 why? 버젼 안맞추고 올려서
    git fetch // 실시
    git branch -a // 어떤 브랜치 있는지 모두 봄
    git merge origin/master // 병합
    여기서 또 오류 발생 -> 충돌해결해야함
    git diff 브랜치명 // 로컬저장소의 브랜치와 원격 브랜치 저장소 사이에 어떤 차이점이 있는지 알아봄(pull 했다면 불가)
    로컬에서 파일 수정
    git commit -a -m “설명”
    git push origin master

추가 참고사항
remote repository : 외부에 저장할 수 있는 저장소
rocal repository : 내컴퓨터에 저장된 저장소

사용 총평 : 기존 svn만 사용해 볼때보다 파일의 변화를 단계별로 버젼으로 기록하여 관리하여 편리하며 앞으로 더 익숙해지면 편할듯하다.
